## What is TOON?

**TOON (Tabular Object-Oriented Notation)** is SochDB's native data format—a compact, schema-aware serialization format designed specifically for LLM context windows.

Think of TOON as "JSON for databases" but optimized for **token efficiency** rather than human readability.

<Info>
TOON achieves **40-67% token reduction** compared to JSON for typical tabular data.
</Info>

## Format Specification

TOON uses a simple, predictable structure:

```ebnf
document     ::= table_header newline row*
table_header ::= name "[" count "]{" fields "}:"
name         ::= identifier
count        ::= integer
fields       ::= field ("," field)*
field        ::= identifier
row          ::= value ("," value)* newline
value        ::= null | bool | number | string | array | ref
```

### Example

```text
users[3]{id,name,email}:
1,Alice,alice@example.com
2,Bob,bob@example.com
3,Charlie,charlie@example.com
```

**Breakdown:**
- `users` = table name
- `[3]` = row count
- `{id,name,email}` = field names (schema)
- `:` = header terminator
- Following lines = data rows (CSV-style)

## Token Comparison: TOON vs JSON

Let's compare the same dataset in both formats:

<CodeGroup>
```json JSON (156 tokens)
[
  {"id": 1, "name": "Alice", "email": "alice@example.com"},
  {"id": 2, "name": "Bob", "email": "bob@example.com"},
  {"id": 3, "name": "Charlie", "email": "charlie@example.com"}
]
```

```text TOON (52 tokens) — 67% reduction!
users[3]{id,name,email}:
1,Alice,alice@example.com
2,Bob,bob@example.com
3,Charlie,charlie@example.com
```
</CodeGroup>

### Why the Difference?

JSON repeats field names in every object:
```json
{"id": 1, "name": "Alice", "email": "alice@example.com"}
{"id": 2, "name": "Bob", "email": "bob@example.com"}
{"id": 3, "name": "Charlie", "email": "charlie@example.com"}
     ↑         ↑              ↑
  Repeated  Repeated       Repeated
```

TOON declares the schema once in the header:
```text
users[3]{id,name,email}:  ← Schema declared once
1,Alice,alice@example.com  ← Just values
2,Bob,bob@example.com      ← Just values
3,Charlie,charlie@example.com
```

<Note>
The larger the dataset, the more dramatic the savings. With 1,000 rows, JSON wastes 999 copies of the field names!
</Note>

## TOON Value Types

TOON supports rich data types with compact syntax:

| Type | TOON Syntax | Example | Notes |
|------|-------------|---------|-------|
| **Null** | `∅` or `null` | `∅` | Minimal representation |
| **Boolean** | `T` / `F` or `true`/`false` | `T` | Single character for compactness |
| **Integer** | number | `42`, `-17` | Standard numeric |
| **Float** | decimal | `3.14159` | Standard decimal |
| **String** | text or `"quoted"` | `Alice`, `"hello, world"` | Quotes only when needed |
| **Array** | `[item;item;...]` | `[1;2;3]` | Semicolon-separated |
| **Reference** | `@table:id` | `@users:42` | Cross-table references |
| **Binary** | `0x...` | `0x48656c6c6f` | Hex-encoded |

### String Quoting Rules

TOON only quotes strings when necessary (spec §7.3):

```rust
// No quotes needed
Alice
Bob
hello_world

// Quotes required (contains comma)
"hello, world"

// Quotes required (looks like number)
"123"
"true"

// Quotes required (special chars)
"multi\nline"
"path/to/file"
```

<Info>
Smart quoting saves tokens. Most database values (names, IDs, simple text) don't need quotes.
</Info>

## TOON in Code

### Rust API

```rust
use sochdb_core::{SochValue, SochSchema, SochTable, SochRow, SochType};

// Define schema
let schema = SochSchema::new("users")
    .field("id", SochType::UInt)
    .field("name", SochType::Text)
    .field("email", SochType::Text)
    .primary_key("id");

// Create table
let mut table = SochTable::new(schema);

// Add rows
table.push(SochRow::new(vec![
    SochValue::UInt(1),
    SochValue::Text("Alice".into()),
    SochValue::Text("alice@example.com".into()),
]));

table.push(SochRow::new(vec![
    SochValue::UInt(2),
    SochValue::Text("Bob".into()),
    SochValue::Text("bob@example.com".into()),
]));

// Format as TOON
let toon_output = table.format();
println!("{}", toon_output);
// Output:
// users[2]{id,name,email}:
// 1,Alice,alice@example.com
// 2,Bob,bob@example.com
```

### Parsing TOON

```rust
use sochdb_core::SochTable;

let toon_input = r#"users[2]{id,name,email}:
1,Alice,alice@example.com
2,Bob,bob@example.com"#;

let table = SochTable::parse(toon_input)?;
assert_eq!(table.schema.name, "users");
assert_eq!(table.rows.len(), 2);
```

## Use Cases

### 1. LLM Context Assembly

**Problem**: You need to send database results to an LLM but have a 4,096 token limit.

```rust
use sochdb::ContextQueryBuilder;

let context = ContextQueryBuilder::new()
    .with_budget(4096)  // Token budget
    .section("RECENT_ORDERS", 0)
        .query("SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 20")
        .format(ContextFormat::Soch)  // Use TOON format
        .done()
    .execute()?;

println!("Tokens used: {}/{}", context.token_count, 4096);
```

TOON typically fits **2-3× more rows** in the same token budget compared to JSON.

### 2. RAG Retrieval Results

```rust
// Vector search returns documents in TOON format
let results = db.hybrid_search(
    vector: query_embedding,
    text_query: "database performance",
    k: 10,
    format: "toon"  // Compact output
)?;

// Results:
// documents[10]{id,title,score,snippet}:
// 42,"Optimizing Queries",0.89,"Use indexes..."
// 17,"Performance Guide",0.85,"Batch operations..."
// ...
```

### 3. Agent Memory Serialization

Store conversation history compactly:

```rust
messages[5]{role,content,timestamp}:
user,"What's the weather?",1704067200
assistant,"Let me check...",1704067201
tool,"Sunny, 72°F",1704067202
assistant,"It's sunny and 72°F",1704067203
user,"Should I bring an umbrella?",1704067210
```

<Note>
For a 100-message conversation, TOON uses ~60% fewer tokens than JSON—saving you API costs and fitting more context.
</Note>

## Advanced Features

### Cross-Table References

TOON supports references between tables:

```text
orders[2]{id,user_id,product_id,quantity}:
1,@users:42,@products:101,2
2,@users:17,@products:205,1

users[2]{id,name}:
42,Alice
17,Bob

products[2]{id,name,price}:
101,"Laptop",999.00
205,"Mouse",25.00
```

### Nullable Fields

Use `∅` for null values:

```text
employees[3]{id,name,manager_id}:
1,Alice,∅
2,Bob,1
3,Charlie,1
```

### Binary Data

```text
files[1]{id,name,content}:
1,"logo.png",0x89504e470d0a1a0a...
```

## Schema Evolution

TOON headers include schema information, making version migration straightforward:

```text
# Version 1
users[2]{id,name}:
1,Alice
2,Bob

# Version 2 (added email field)
users[2]{id,name,email}:
1,Alice,alice@example.com
2,Bob,bob@example.com
```

Parsers can detect schema changes by comparing field lists.

## Performance Characteristics

| Operation | Complexity | Notes |
|-----------|-----------|-------|
| **Serialize** | O(n) | Linear scan, minimal allocations |
| **Parse** | O(n) | Single-pass parser |
| **Token count** | ~40-67% less | Compared to JSON |
| **Memory** | Same as data | No intermediate structures |

## Comparison with Other Formats

| Format | Tokens (1K rows) | Human-Readable | Schema Info | Binary Support |
|--------|-----------------|----------------|-------------|----------------|
| **JSON** | 100% (baseline) | ✅ Excellent | ❌ No | ❌ Base64 only |
| **CSV** | ~60% | ⚠️ Limited | ❌ No header | ❌ No |
| **TOON** | **~40%** | ✅ Good | ✅ In header | ✅ Hex encoding |
| **Protobuf** | ~30% | ❌ Binary | ✅ Separate .proto | ✅ Native |
| **MessagePack** | ~35% | ❌ Binary | ❌ No | ✅ Native |

<Info>
TOON balances token efficiency with readability. It's more compact than JSON/CSV but remains text-based and debuggable.
</Info>

## Best Practices

### When to Use TOON

✅ **Use TOON for:**
- LLM context assembly (maximize data per token)
- RAG retrieval results (compact ranked lists)
- Agent memory serialization (conversation history)
- Database query results sent to LLMs

❌ **Don't use TOON for:**
- Public APIs (use JSON for compatibility)
- Human-edited config files (use JSON/YAML)
- Deeply nested objects (JSON is clearer)
- When you need comments (use YAML)

### Optimization Tips

1. **Order fields by size**: Put large text fields last
2. **Use short field names**: `id` not `identifier`
3. **Batch similar queries**: One table with 100 rows > 100 separate values
4. **Leverage references**: `@users:42` instead of embedding full user object
