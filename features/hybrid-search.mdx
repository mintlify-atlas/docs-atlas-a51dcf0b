## Overview

SochDB's hybrid search combines the best of both worlds: semantic vector similarity and traditional keyword matching. By fusing results from HNSW vector search and BM25 lexical search, you get more accurate retrieval than either method alone.

<Note>
**Why Hybrid Search?** Vector search alone can miss exact keyword matches. Keyword search alone can't understand semantic meaning. Hybrid search combines both to maximize recall and precision.
</Note>

## Key Features

- **HNSW Vector Search**: Fast approximate nearest neighbor search with configurable accuracy
- **BM25 Keyword Search**: Classical information retrieval with term frequency and inverse document frequency
- **Reciprocal Rank Fusion (RRF)**: Robust score fusion algorithm that combines rankings from multiple sources
- **Metadata Filtering**: Pre-filter candidates by metadata before search (no post-filtering)
- **Configurable Weights**: Control the balance between semantic and lexical signals

```python Advanced Configuration
from sochdb import HybridQuery, FusionMethod

# Build hybrid query with advanced options
query = (
    HybridQuery(collection)
    .with_vector(query_embedding, weight=0.7)
    .with_lexical("database performance optimization", weight=0.3)
    .filter_eq("status", "published")
    .filter_range("score", min=0.8, max=None)
    .with_fusion(FusionMethod.RRF)  # Reciprocal Rank Fusion
    .with_rrf_k(60)  # RRF smoothing parameter
    .limit(20)
    .min_score(0.5)
)

results = query.execute()
```

## Reciprocal Rank Fusion (RRF)

SochDB uses RRF as the default fusion algorithm because it's robust to score distribution differences between vector and lexical search.

### Formula

For each document `d`, the RRF score is:

```
score(d) = Σ w_i / (k + rank_i(d))
```

Where:
- `w_i` is the weight for retrieval system `i` (e.g., 0.7 for vector, 0.3 for BM25)
- `rank_i(d)` is the rank of document `d` in system `i`'s results (0-indexed)
- `k` is a smoothing constant (default: 60)

### Why RRF?

<Tabs>
  <Tab title="Advantages">
    - **Score-independent**: Works with incomparable scores (vector distances vs BM25)
    - **Robust**: Not affected by outliers or score distribution skew
    - **Simple**: One parameter (`k`) to tune
    - **Proven**: Used by major search systems (Elasticsearch, OpenSearch)
  </Tab>
  <Tab title="vs Weighted Sum">
    Weighted sum requires normalized scores:
    ```
    score(d) = w1 * normalize(vector_score) + w2 * normalize(bm25_score)
    ```
    
    Problems:
    - Score distributions differ widely (cosine: 0-1, BM25: unbounded)
    - Normalization can amplify noise
    - Sensitive to outliers
    
    RRF uses **ranks**, not scores, avoiding these issues.
  </Tab>
  <Tab title="Tuning k">
    The `k` parameter controls how much weight is given to lower ranks:
    
    - **Low k (e.g., 10)**: Top results matter more, lower ranks matter less
    - **High k (e.g., 100)**: More equal weighting across ranks
    - **Default k=60**: Robust across most datasets
    
    ```python
    # Favor top results
    results = collection.hybrid_search(
        vector=query_embedding,
        text_query="query",
        k=10,
        alpha=0.7,
        rrf_k=10  # Lower k
    )
    ```
  </Tab>
</Tabs>

## Configuration Options

### Alpha Parameter (Vector/Keyword Balance)

```python
# Pure vector search (alpha=1.0)
results = collection.hybrid_search(
    vector=query_embedding,
    text_query="machine learning",
    k=10,
    alpha=1.0  # 100% vector, 0% keyword
)

# Balanced hybrid (alpha=0.5)
results = collection.hybrid_search(
    vector=query_embedding,
    text_query="machine learning",
    k=10,
    alpha=0.5  # 50% vector, 50% keyword
)

# Keyword-heavy (alpha=0.3)
results = collection.hybrid_search(
    vector=query_embedding,
    text_query="machine learning",
    k=10,
    alpha=0.3  # 30% vector, 70% keyword
)
```

<Note>
**Choosing Alpha**: Use high alpha (0.7-0.9) for semantic queries. Use low alpha (0.3-0.5) for queries with specific keywords or product codes.
</Note>

### Metadata Filtering

Filters are applied **before** search, not after (pre-filtering):

```python
results = collection.hybrid_search(
    vector=query_embedding,
    text_query="database",
    k=10,
    alpha=0.7,
    filters={
        # Equality
        "category": "tech",
        "status": "published",
        
        # Range
        "year": {"$gte": 2020, "$lte": 2024},
        "score": {"$gt": 0.8},
        
        # Array contains
        "tags": {"$contains": "python"},
        
        # In set
        "author": {"$in": ["alice", "bob", "charlie"]}
    }
)
```

<Warning>
**Pre-filtering Only**: SochDB enforces a security invariant: all filtering happens during candidate generation. There is no post-filtering. This guarantees:
- No leakage of filtered documents
- Consistent result counts
- Deterministic security boundaries
</Warning>

## Use Cases

<AccordionGroup>
  <Accordion title="Technical Documentation Search">
    Combine semantic understanding with exact code/API matches:
    ```python
    # User searches: "authentication error handling"
    results = collection.hybrid_search(
        vector=embed("authentication error handling"),
        text_query="auth error AuthenticationError",
        k=10,
        alpha=0.6  # Balanced for both semantic + exact matches
    )
    ```
    **Why hybrid?** Finds both conceptually related docs ("login issues") and exact API references ("AuthenticationError").
  </Accordion>
  
  <Accordion title="E-commerce Product Search">
    Find semantically similar products while matching SKUs/brands:
    ```python
    # User searches: "wireless headphones sony"
    results = collection.hybrid_search(
        vector=embed("wireless headphones"),
        text_query="wireless bluetooth sony WH-1000XM5",
        k=20,
        alpha=0.4,  # Lower alpha for keyword matching (SKUs, brands)
        filters={"in_stock": True}
    )
    ```
    **Why hybrid?** Vector search finds similar headphones, BM25 boosts exact brand/model matches.
  </Accordion>
  
  <Accordion title="Customer Support Tickets">
    Match similar issues while finding specific error codes:
    ```python
    # Support ticket: "getting error 504 when uploading files"
    results = collection.hybrid_search(
        vector=embed("upload timeout error"),
        text_query="504 gateway timeout upload",
        k=10,
        alpha=0.5,
        filters={"status": "resolved"}

    ```
    **Why hybrid?** Vector search finds conceptually similar issues, BM25 matches specific error codes.
  </Accordion>
  
  <Accordion title="Legal/Compliance Document Retrieval">

    **Why hybrid?** Find related regulations (semantic) + exact statute citations (BM25).

## Performance

From SochDB benchmarks (50K documents, 768D embeddings, Apple M1 Ultra):

| Metric | SochDB Hybrid | Pure Vector | Pure BM25 |
|--------|--------------|-------------|----------|
| **Recall@10** | **95.2%** | 89.1% | 78.3% |
| **Precision@10** | **0.92** | 0.85 | 0.71 |
| **Latency (p50)** | **4.1ms** | 3.3ms | 2.8ms |
| **Latency (p99)** | **6.8ms** | 5.9ms | 4.2ms |

<Note>
Hybrid search latency is only ~24% higher than pure vector search, while recall improves by 7%. The trade-off is well worth it for most applications.
</Note>

## Best Practices

<CardGroup cols={2}>
  <Card title="Choose Alpha Wisely" icon="balance-scale">
    - **0.7-0.9**: Conceptual queries ("how to optimize database")
    - **0.5-0.6**: Balanced queries ("python logging best practices")
    - **0.3-0.4**: Exact match priority ("error code 404", "SKU-12345")

  <Card title="Index Content Fields" icon="file-lines">
    Set `content_field` to the primary text field for BM25:
    ```python

        name="docs",

        enable_hybrid_search=True,
        content_field="text"  # Main content
    )
    ```
  </Card>
  
  <Card title="Use Pre-filtering" icon="filter">
    Apply filters during search, not after:
    ```python
    # Good: Pre-filter
    results = collection.hybrid_search(
        vector=v, text_query=q,
        filters={"status": "active"}
    )
    
    # Bad: Post-filter (not supported)
    # results = [r for r in all_results
    ```

  </Card>
  
  <Card title="Tune RRF k" icon="sliders">
    Default k=60 works for most cases. Adjust if:
    - **k=10-30**: When top results are much more important
    - **k=80-100**: When you want more uniform weighting
    
    Test on your dataset to find optimal value.
  </Card>

## Advanced: Multi-field BM25

Search across multiple text fields with different weights:

```python
from sochdb import HybridQuery

query = (
    HybridQuery(collection)
    .with_vector(query_embedding, weight=0.6)
    .with_lexical_fields(
        "database performance",
        weight=0.4,
        fields=["title", "content", "tags"]  # Search multiple fields
    )
    .limit(10)
)

results = query.execute()
```
